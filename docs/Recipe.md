# Recipe 정리 가이드

> 보안 진단 스크립트에 사용할 텍스트/멘트를 정리하는 범용 가이드

**상위 문서**: [USG.md](./USG.md)

---

## 1. 개요

본 가이드는 보안 진단 스크립트 작성 시 사용할 텍스트/멘트를 정리하는 방법을 정의합니다.
모든 플랫폼(OS, DB, WAS 등)의 recipe 파일은 이 가이드를 기반으로 작성합니다.

---

## 2. Recipe의 목적

- 스크립트 작성 시 참고할 수 있는 표준화된 텍스트/멘트 제공
- 점검 항목의 의미, 목적, 방법을 명확히 문서화
- 일관성 있는 문서 구조 유지
- 스크립트 개발자와 평가자가 공통으로 참고할 수 있는 자료

---

## 3. 파일 구조

### 3.1 디렉토리 구조

```
scripts/{플랫폼 경로}/
└── recipe/
    ├── README.md          # 서머리 및 목차
    ├── {PREFIX}-XX.md     # 각 항목별 파일
    └── ...
```

**예시:**
```
scripts/4. PC/recipe/
├── README.md
├── PC-01.md
├── PC-02.md
└── ...

scripts/1. OS/Linux/RHEL/recipe/
├── README.md
├── U-01.md
├── U-02.md
└── ...
```

### 3.2 파일명 규칙

- **형식**: `{PREFIX}-XX.md`
  - `{PREFIX}`: 플랫폼/항목 코드 접두사 (예: U, PC, W, M, O, S, A, N, T, I)
  - `XX`: 2자리 숫자 (01, 02, 03, ...)
- **예시**:
  - `U-01.md` (Linux)
  - `PC-01.md` (Windows PC)
  - `W-01.md` (Windows Server)
  - `M-01.md` (MySQL)
  - `O-01.md` (Oracle)

## 5. 섹션별 상세 설명

### 5.1 기본 정보

**목적**: 항목의 기본 식별 정보 제공

**포함 항목**:
- **항목코드**: `{PREFIX}-XX` 형식 (예: U-01, PC-01, W-01)
- **중요도**: 상/중/하
- **카테고리**: 점검 항목의 분류 카테고리
- **항목명**: 점검 항목의 전체 이름

**예시**:
```markdown
## 기본 정보

- **항목코드**: U-01
- **중요도**: 상
- **카테고리**: 1. 계정 관리
- **항목명**: root 계정 원격 접속 제한
```

### 5.2 개요

**목적**: 점검 항목에 대한 기본적인 이해 제공

**하위 섹션**:
- **점검 내용**: 무엇을 점검하는지 간략히 설명
- **점검 목적**: 왜 점검하는지 설명 (보안 목적, 위험 방지 등)

### 5.3 보안 위협

**목적**: 해당 취약점으로 인한 보안 위협 설명

**포함 내용**:
- 취약점이 존재할 경우 발생할 수 있는 위험
- 공격 시나리오
- 잠재적 영향 범위

### 5.4 참고

**목적**: 관련 용어, 기술적 배경 정보 제공

**포함 내용**:
- 관련 용어 설명
- 기술적 배경 정보
- 각주나 보충 설명
- 관련 표준, 규정 등

### 5.5 점검 대상 및 판단 기준

**목적**: 점검 대상과 판단 기준을 명확히 정의

**하위 섹션**:
- **대상**: 점검 대상 플랫폼/OS/버전 등
- **판단 기준**: 
  - **양호**: 점검 통과 조건
  - **취약**: 점검 실패 조건
  - 명확하고 구체적으로 작성 (모호한 표현 지양)

### 5.6 점검 방법

#### 5.6.1 자동 점검 (스크립트)
- 스크립트로 자동 점검이 가능한 경우
- 점검에 사용할 명령어, 코드
- 코드 블록으로 작성

#### 5.6.2 수동 점검
- 수동 점검이 필요한 경우
- 플랫폼/OS별로 구분하여 작성
- 단계별로 명확히 작성

### 5.7 조치 시 영향

**목적**: 조치 시 예상되는 영향 설명

### 5.8 우려사항 (선택)

**목적**: 스크립트 작성 시 또는 실제 적용 시 고려사항 제시

### 5.9 솔루션 (선택)

**목적**: 우려사항에 대한 해결 방안 제시

## 6. 점검 방법 분류 기준

### 6.1 자동 점검

다음과 같은 경우 **자동 점검**으로 분류합니다:

- 스크립트로 자동으로 점검이 가능한 경우
- 명령어 실행으로 현재 상태 확인 가능
- 파일 내용 읽기로 설정 확인 가능
- 서비스 상태, 프로세스 상태 확인 등

### 6.2 수동 점검

다음과 같은 경우 **수동 점검**으로 분류합니다:

- 내부 정책, 문서 확인 필요
- 관리자 인터뷰 필요
- 스크립트만으로는 판단 불가능한 경우

## 7. 코드 블록 사용 규칙

### 7.1 언어 지정

코드 블록은 해당 플랫폼/언어에 맞게 언어를 지정합니다:

| 플랫폼 | 언어 태그 |
|--------|----------|
| Linux/Unix | `bash`, `sh` |
| Windows | `powershell`, `cmd`, `batch` |
| MySQL | `sql`, `mysql` |
| Oracle | `sql`, `oracle` |
| 일반 설정 파일 | `ini`, `conf`, `yaml`, `json` |

### 7.2 명령어 작성 규칙

- 실제 실행 가능한 명령어로 작성
- 변수, 플레이스홀더는 명확히 표시 (예: `<파일경로>`, `<서비스명>`)
- 주석을 통해 설명 추가 가능
- 관리자 권한 필요한 경우 명시

## 8. README.md 작성 규칙

recipe 디렉토리의 `README.md`는 다음 구조를 가집니다:

```markdown
# {플랫폼명} Recipe

{플랫폼} 취약점 진단 스크립트에 사용할 텍스트/멘트 모음집

## 서머리

| 코드 | 위험도 | 항목명 | 점검 방법 |
|------|--------|--------|-----------|
| {PREFIX}-01 | 상 | {항목명} | 자동 |
| {PREFIX}-02 | 중 | {항목명} | 수동 |
| ... | ... | ... | ... |

## 목차

### {카테고리명}
- [{PREFIX}-01: {항목명}](./{PREFIX}-01.md)
- [{PREFIX}-02: {항목명}](./{PREFIX}-02.md)

## 참고

- 원본: {원본 가이드 출처}
- 대상: {대상 플랫폼/OS/버전}
```

## 10. 작성 체크리스트

recipe 파일을 작성할 때 다음 항목을 확인합니다:

### 필수 항목
- [ ] 기본 정보 (항목코드, 중요도, 카테고리, 항목명)
- [ ] 개요 (점검 내용, 점검 목적)
- [ ] 보안 위협
- [ ] 점검 대상 및 판단 기준
- [ ] 점검 방법 (자동/수동)
- [ ] 조치 시 영향

### 선택 항목
- [ ] 참고 (필요한 경우)
- [ ] 우려사항 (필요한 경우)
- [ ] 솔루션 (우려사항이 있는 경우)

### 품질 확인
- [ ] 마크다운 형식 올바르게 사용
- [ ] 코드 블록에 적절한 언어 태그 지정
- [ ] 명령어가 실제로 실행 가능한 형태
- [ ] 판단 기준이 명확하고 구체적
- [ ] 플랫폼/OS별 구분이 명확
